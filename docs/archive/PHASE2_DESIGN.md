# claw-rl Phase 2: OpenClaw-RL 核心实现

**目标：** 实现从用户对话中自动提取学习信号并改进

---

## 🎯 核心设计

### OpenClaw-RL 核心洞察

**Every agent interaction produces a next-state signal that encodes how the agent performed.**

每个 Agent 交互都会产生 **next-state signal**（下一状态信号）：
- 用户回复
- 工具执行结果
- 终端/GUI 状态变化

**两种信息：**
| 信号类型 | 含义 | 提取方式 |
|---------|------|---------|
| **Evaluative**（评估） | 动作执行得如何 | PRM Judge → 标量奖励 (+1/-1/0) |
| **Directive**（指令） | 动作应该如何不同 | OPD → Token 级方向监督 |

## 📦 组件 1: PRM Judge

### 功能
评估 Friday 的动作质量，输出标量奖励 r ∈ {+1, -1, 0}

### 判断规则

| 用户回复类型 | 奖励 | 说明 |
|------------|------|------|
| 明确表扬 ("好的"、"谢谢"、"很好") | +1 | 满意 |
| 明确纠正 ("不对"、"应该是"、"错了") | -1 | 不满意 |
| 重新提问同样问题 | -1 | 对答案不满意 |
| 继续追问新问题 | +1 | 对答案满意，继续对话 |
| 无明显信号 | 0 | 中性 |

### 实现方式

**方案 A: 规则判断（简单快速）**
```bash
# 基于关键词匹配
if reply contains "谢谢" or "好的" or "很好" → +1
if reply contains "不对" or "错了" or "应该" → -1
```

**方案 B: LLM Judge（更准确）**
```
Prompt:
"判断用户回复是否表示对 AI 助手的回答满意：
用户回复：{reply}
选项：满意 (+1) / 不满意 (-1) / 中性 (0)
输出：+1/-1/0"
```

## 📦 组件 2: Binary RL

### 功能
将评估信号转化为学习记录，用于后续提示优化

### 实现流程

```
动作 at + 下一状态 st+1 → PRM Judge → r ∈ {+1, -1, 0}
                                    ↓
                          记录到 .rewards/{date}.jsonl
                                    ↓
                          累积奖励 → 触发记忆更新
```

### 奖励记录格式

```json
{
  "timestamp": "2026-03-17T17:00:00+08:00",
  "session_id": "session_12345",
  "turn_id": 5,
  "action": "文件创建：/Users/liantian/workspace/test.md",
  "next_state": "用户回复：不对，应该放到 ~/.openclaw/workspace/",
  "reward": -1,
  "reason": "目录选择错误"
}
```

### 累积奖励触发学习

```
当同一场景累积奖励 < -3 时：
→ 生成学习条目到 .learnings/LEARNINGS.md
→ 更新提示模板，强化该场景的正确行为
```

## 📦 组件 3: OPD (On-Policy Distillation)

### 功能
从用户纠正中提取指令信号，转化为 Token 级方向监督

### 核心洞察

用户说"你应该先检查文件再编辑"不仅表示错了（evaluative），还指出了如何改正（directive）。

### 实现流程

```
Step 1: Hint Extraction
  Judge(at, st+1) → {score ∈ {+1,-1}, hint ∈ T*}
  
  如果用户纠正包含具体指示：
  "不对，应该先检查文件" → hint = "操作前先检查目标文件是否存在"

Step 2: Hint Selection
  选择最具体、最可操作的 hint
  过滤掉模糊的反馈（如"不好"）

Step 3: Enhanced Context
  将 hint 添加到会话前注入的上下文中：
  
  📌 学习到的改进方向：
  - 用户反馈：应该先检查文件
  - 改进行为：操作前执行存在性检查

Step 4: Token-Level Guidance
  在类似场景下，优先使用 hint 中的关键词和短语
```

### Hint 提取规则

| 用户反馈类型 | 提取的 Hint |
|------------|-----------|
| "应该先 X" | "操作前先 X" |
| "不要 Y" | "避免 Y" |
| "用 Z 方法" | "优先使用 Z 方法" |
| "太长了/短了" | "调整回复长度为中等" |

## 📦 组件 4: 异步训练循环

### 功能
后台处理学习信号，更新记忆和提示

### 实现架构

```
┌─────────────────────────────────────────────────────────┐
│              Asynchronous Learning Loops                │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  ┌──────────┐    ┌──────────┐    ┌──────────┐        │
│  │  Serving │    │ Collect  │    │  Train   │        │
│  │  (在线)  │    │ (后台)   │    │ (后台)   │        │
│  │          │    │          │    │          │        │
│  │ 响应用户 │ →  │ 收集信号 │ →  │ 更新记忆 │        │
│  │ 请求     │    │ 判断奖励 │    │ 优化提示 │        │
│  └──────────┘    └──────────┘    └──────────┘        │
│                                                         │
│  三个循环独立运行，无阻塞依赖！                         │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

### 实现方式

由于我们是文本 Agent，简化为：

1. **Serving:** Friday 正常响应用户
2. **Collect:** 后台脚本定期处理新对话
3. **Train:** 更新 .learnings/ 和提示模板
